Overview

In this project, we explored the building blocks of geometric modeling, including implementing Bezier surfaces and adding functionality to triangle meshes. In task 1, we learned how to create a curve from any number of anchor points by implementing Bezier curves using de Casteljau’s algorithm. We were able to extend the Bezier curves and create Bezier surfaces in task 2 by using multiple iterations of de Casteljau’s.

In Section 2, we learned how to implement area-weighted vertex normals (task 3), which allow for better shading on smooth objects. We then added features to our triangle mesh including edge flips (task 4) and edge splits (task 5). Finally, using edge flips and edge splits, we implemented loop subdivision (task 6), which allows for the triangle mesh to subdivide and create a mesh with much smaller triangles, allowing for mesh that is more detailed and smooth. In the end, we could take triangle meshes and flip or split edges as we wished, and also perform multiple iterations of loop subdivision.

In this project, for section 2 in particular, we became very familiar with triangle mesh objects (vertices, edges, halfedges, and faces). We learned how to systematically move through different triangle elements with halfedges, and learned that being overly careful is always a better approach when dealing with a lot of elements that change after a mesh operation.

Part 1: Bezier Curves with 1D de Casteljau Subdivision

Briefly explain de Casteljau's algorithm and how you implemented it in order to evaluate Bezier curves.

To obtain all the points on a Bezier curve, we compute individual points across a parameter t∈[0,1]t \in [0, 1]t∈[0,1]﻿. For a given value of ttt﻿ we can evaluate 1 point on the Bezier curve with de Casteljau's algorithm. de Casteljau's algorithm uses linear interpolation between pairs of adjacent points and outputs sublevels of new points. At the highest level, we are given the nnn﻿ control points of the Bezier curve. With nnn﻿ points, we interpolate n−1n-1n−1﻿ pairs of adjacent points to obtain n−1n-1n−1﻿ points that define the next level. All linear interpolations are performed with the weights ttt﻿ and 1−t1-t1−t﻿ for the 2 points interpolating between. This process is recursively done until there is only 1 point remaining (interpolated from the last level with 2 points), which is a point on the Bezier curve for those original nnn﻿ control points, linearly interpolated with the initial parameter ttt﻿. By doing this step for all ttt﻿ ∈\in∈﻿ [0,1][0,1][0,1]﻿, we can use de Casteljau's algorithm to get all points that lie on the Bezier curve, thus creating our curve.

Part 2: Bezier Surfaces with Separable 1D de Casteljau

Briefly explain how de Casteljau algorithm extends to Bezier surfaces and how you implemented it in order to evaluate Bezier surfaces.

For Bezier surfaces, our surface is now in 3 dimensions instead of solely based on 2-dimensional curves. Instead of a single curve, we want to interpolate over an entire surface using a grid of n×nn \times nn×n﻿ 3D control points. In other words, we want to parametrize and find the value at any (u,v)(u,v)(u,v)﻿ on the surface where u,v∈[0,1]u, v \in [0,1]u,v∈[0,1]﻿. We can still use the de Casteljau algorithm with a few extra steps to extend to Bezier surfaces. To do this, we can consider the points in each row independently and define a Bezier curve for each row, giving us nnn﻿ parallel rows in 1 direction (axis) of the grid. Applying de Casteljau’s algorithm for each row, we then parametrize by uuu﻿ and find 1 point on each Bezier curve, giving nnn﻿ new control points. These nnn﻿ control points can then define a new “moving” Bezier curve in the other direction (other axis). Applying de Casteljau’s algorithm again, we parametrize on this curve by vvv﻿ to find 1 point, which is the interpolated surface point at (u,v)(u,v)(u,v)﻿.

Part 3: Area-Weighted Vertex Normals

Briefly explain how you implemented the area-weighted vertex normals.

To compute the area-weighted vertex normal, we computed the area-weighted normals of the faces surrounding the vertex (the neighboring triangles) and then normalized the sum of those area-weighted normals of the faces.

To compute the area-weighted normals, we iterated through the faces surrounding the vertex and used the normal() function associated with the face class, which uses a cross-product of 2 vertices of the triangular face to compute the area-weighted normal.

Finally, to normalize, we divided the sum of the weighted normals by the norm of that sum.

Part 4: Edge Flip

Briefly explain how you implemented the edge flip operation and describe any interesting implementation / debugging tricks you have used.

The edge flip operation entails converting the edge shared between 2 triangles into a different edge orientation to produce a pair of 2 different triangles as in the diagram above.

Implementing edge flip did not require the creation of any new mesh features. The main changes to the mesh to “flip” the edge was taking the feature corresponding to the edge being flipped and reassigning the vertices pertaining to its 2 halfedges. Likewise the vertex objects the un-flipped edge joined needed to be updated to ensure they did not point to the wrong halfedges.

For consistency and to avoid any errors with mesh objects with invalid pointers, we first gathered all faces, vertices, edges, and halfedges from the two triangles before the flip. We then set all the pointers for each of these mesh objects for their state after the flip, even for those which technically are not affected by the edge flip. The relationships between these objects are seen in the diagrams above. For consistency in labeling, all halfedges of a face are ordered in a clockwise orientation, all halfedges have a pointer to the vertex that is the "source" of the halfedge, and a vertex only has a pointer to one of the halfedges that points away from the vertex.

We made sure to check if an edge was a boundary edge, in which we did not perform the edge flip operation. Our edge flip operation does a constant amount of work because we always only involve the elements from the 2 triangles around the edge being flipped.

Part 5: Edge Split

Briefly explain how you implemented the edge split operation and describe any interesting implementation / debugging tricks you have used.

The edge split operation entails adding a new vertex at the center of an edge and then connecting the new vertex to each opposing vertex to generate 2 new triangle faces as seen above.

Implementing an edge split requires the creation of a few new mesh features and updating existing mesh features. First, a new vertex object is created centered at the target edge. Next, the existing edge was split into a new edge across the vertex, so this necessitated a new edge and 2 new halfedges. Next, in the perpendicular direction, there are 2 new edges and correspondingly, 4 new halfedges. Lastly, with the new edges, there are 2 new faces. For all remaining mesh features, we updated pointers accordingly with the newly created features.

Our implementation approach was similar to that of edge flip, where we explicitly gathered all mesh objects before the edge split and then set all the correct pointers after the edge flip (and new element creation). These relationships are described in the diagrams above. We continue following the same conventions as in edge flip: all halfedges of a face are ordered in a clockwise orientation, all halfedges have a pointer to the vertex that is the "source" of the halfedge, and a vertex only has a pointer to one of the halfedges that points away from the vertex.

Part 6: Loop Subdivision for Mesh Upsampling

Briefly explain how you implemented the loop subdivision and describe any interesting implementation / debugging tricks you have used.

The loop subdivision algorithm enables us to upsample the mesh by dividing existing triangular faces into 4 smaller triangles (4-1 subdivision). This process utilizes our previously implemented edge split and edge flip operations together. At a high level there are 2 main steps. First is subdividing all triangles using edge splits and correcting edges with edge flips (shown below).

The second step is to set the correct positions of new vertices added as well as to update the positions of existing vertices. This process is done via a weighting scheme shown in the diagram below. New vertices are created along edges that are split. The positions are weighted as 38\frac{3}{8}83​﻿ of the positions of vertices along the old edge where the new vertex is created and 18\frac{1}{8}81​﻿ of the positions of the remaining 2 vertices. Old vertices are adjusted based on the degree of the vertex as shown in the diagram.

For our actual implementation in code, these steps were encoded over 5 parts:

  1. New positions of the old vertices were computed and stored in the vertex structure but not yet assigned as the old positions are needed to compute the positions of new vertices (computed according to the weighting scheme).
  1. The correct locations of new vertices are also calculated and stored in the edge structure from which they will be calculated (according to the weighting scheme below). These vertices are not yet created.
  1. All edges are split and the newly created vertices are assigned their computed positions from step 2. These vertices are also labeled as new vertices to distinguish them from old ones.
  1. To get the final desired mesh design, corrections are applied by flipping any new edges that connect old and new vertices. The labeling of old and new vertices in the vertex structure was used here, and we did not flip any “new” edges that were halves of edges in the original mesh.
  1. The positions of all old vertices are updated according to calculations in step 1.

Most of the implementation process occurred smoothly but we did encounter a few bugs.

When we had initially implemented the loop subdivision, during the step 4, where we were flipping edges that were between a new vertex and an old vertex, we had not labeled the edges that were “new” from an existing edge that got split into two as as “old” edge, so we were flipping too many edges. Upon further inspection of the split and flip diagram above, we realized that we should have skipped over those edges when flipping edges in our mesh.

Another bug that arose was not using floating point numbers in all computations. When some numbers were taken as integers, fractions rounded down to zero and all new vertex positions were calculated to be (0,0,0)(0, 0, 0)(0,0,0)﻿, causing the mesh to “collapse in on itself.” We remedied the situation by making sure all arithmetic was done with floats.
